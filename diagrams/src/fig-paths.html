Two delivery paths
Secured with — and without — the starters
The same secure-by-default resource server, whether it arrives transitively via the application starter or is added directly.
Fireflysecurity
With the application starter
secure-by-default · zero security code
<dependency> <groupId>org.fireflyframework</groupId> <artifactId>fireflyframework-starter-application</artifactId> </dependency>
▸
Pulls security-resource-server + method-policy transitively.
▸
Every reactive web app is default-deny out of the box.
▸
JWT validation, @Secure, hardened headers — already on.
▸
You only declare the public routes (permit-matchers).
Without the starter
standalone resource server
<dependency> <groupId>org.fireflyframework</groupId> <artifactId>fireflyframework-security-resource-server</artifactId> </dependency> <!-- + -method-policy for @Secure -->
▸
Same ResourceServerAutoConfiguration: decoder, default-deny chain, headers.
▸
Conditional on a reactive web app; opt-out with one property.
▸
Configure issuer / audiences for your IdP.
▸
Add security-method-policy for @Secure / @PreAuthorize.
Shared configuration — the only thing most services declare:
firefly.security.resource-server: enabled: true # default for reactive web apps; opt-out switch permit-matchers: [ /actuator/health/**, /public/** ] # the ONLY way a route is public issuer: https://idp.example/realms/firefly audiences: [ firefly-api ] role-claim-paths: [ roles, realm_access.roles, cognito:groups ]
